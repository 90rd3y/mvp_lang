# Система Типов

Этот документ описывает систему типов, правила проверки и приведения типов в учебном языке на основе реальной реализации семантического анализатора (`Semantic.cpp`).

## 1. Строгая Статическая Типизация

Язык использует **строгую статическую типизацию** (Strict Static Typing).
1. Тип каждой переменной, выражения и функции разрешается и проверяется на этапе статического анализа (до выполнения программы).
2. Запрещены любые неявные приведения типов (Implicit Conversions / Coercions).
3. Присваивание, вызовы функций и бинарные операции требуют строгого соответствия типов (реализовано через функцию `types_compatible`).

## 2. Базовые типы (Примитивы)

| Ключевое слово | Назначение |
|----------------|------------|
| `целое`        | Целочисленный знаковый тип. |
| `бцелое`       | Целочисленный беззнаковый тип. |
| `вещ`          | Число с плавающей точкой. |
| `лог`          | Логическое значение (`истина` или `ложь`). |
| `строка`       | Строковый тип. |
| `символ`       | Одиночный символ. |
| `пусто`        | Отсутствие типа (используется как возвращаемое значение функций, не возвращающих данных). |

## 3. Пользовательские типы

### 3.1 Синонимы типов (Type Aliases)
```text
тип Число = целое;
тип Матрица = Число; // Синоним на синоним
тип Тройка = целое[3]; // Синоним может указывать и на тип массива
```
Синонимы типов являются "прозрачными". При любой проверке совместимости семантический анализатор рекурсивно "раскручивает" (unwraps) синонимы вплоть до базового типа (через внутреннюю функцию `resolve_alias`). Таким образом, тип `Число` полностью взаимозаменяем с `целое`. Это в равной степени верно и для синонима массива: `Тройка` взаимозаменяем с `целое[3]`, а два синонима, указывающих на один и тот же по форме массив (например, `тип А = целое[3]; тип Б = целое[3];`), взаимозаменяемы и между собой (см. §3.2 про канонизацию).

### 3.2 Массивы (Static Arrays)
```text
целое[10] массив = [1, 2, 3];
```
Массивы определяются базовым типом и фиксированным размером.
* Два массива считаются совместимыми, только если совпадают их базовые типы (с учётом алиасов) **и их размеры** (`types_compatible`).
* **Канонизация типа массива.** Тип массива — структурный (в отличие от структуры, у него нет собственного имени), поэтому каждое отдельное упоминание `БазовыйТип[N]` в принципе могло бы порождать новый `TypeId`. Чтобы избежать ситуации, когда два места в коде описывают "один и тот же" по смыслу массив, но получают разные `TypeId`, регистрация типа массива всегда идёт через внутренний кэш `Analyzer::array_type_cache: (базовый_тип, размер) → TypeId` — при повторном упоминании того же сочетания (базовый тип после раскрутки алиасов, размер) возвращается уже существующий `TypeId`. Благодаря этому не только `types_compatible`, но и прямое сравнение `TypeId == TypeId` (используемое, например, в проверке бинарных операторов и при сверке сигнатур прототипа/определения функции) корректно распознаёт структурно одинаковые массивы как один тип. В частности, повторное присваивание переменной-массиву нового литерала того же размера (`целое[3] a = [1,2,3]; a = [4,5,6];`) — корректный код, а не статическая ошибка.
* Массив может быть базовым типом **любого** места, где допустим `БазовыйТип[N]`: переменной, поля структуры, параметра функции, типа возврата функции, синонима типа. Во всех случаях базовый тип массива обязан пройти проверку `is_allowed_array_base_type` — допустимы примитивы, `строка` и структуры; массив массивов (многомерные массивы) запрещён синтаксически (парсер не допускает второй `[...]` подряд) и семантически (`is_allowed_array_base_type` не включает сам `TypeKind::Array`).
* Индексация (`arr[i]`) строго требует, чтобы тип выражения индекса был `целое`. Границы индекса не проверяются статически (индекс — рантайм-значение), но проверяются на этапе интерпретации (см. `SEMANTICS.md` §9) — выход за границы даёт управляемую ошибку выполнения, а не неопределённое поведение.
* При инициализации массива через литерал (`[1, 2, 3]`), пустые массивы (`[]`) запрещены. Тип всех последующих элементов в литерале обязан быть `types_compatible` с типом самого первого элемента.
* Многомерные массивы напрямую не поддерживаются в синтаксисе. Массивы аллоцируются в Арене и хранятся как пара "указатель на первый элемент + количество элементов"; последнее используется в рантайме для проверки границ индекса и для поэлементного сравнения `==`/`!=` (см. `SEMANTICS.md` §5.1, §9).

### 3.3 Структуры (Structs)
```text
структура Точка { целое х; целое у; }
```
Каждая структура регистрируется в таблице типов как уникальный идентификатор (Nominal Typing).
Две структуры с одинаковыми именами полей и их типами считаются **несовместимыми**, если это разные структуры.
Поддерживается вложенность структур. Рекурсивное объявление структуры внутри самой себя **по значению** запрещено и вызывает ошибку; поле-массив того же типа, что и сама структура (например, `Узел[2] дети;` внутри `структура Узел`), не считается рекурсивным и разрешено, так как массив хранится как указатель, а не встраивается по значению.

Поле структуры может само быть массивом (`целое[3] поле;`, либо массивом структур/синонимов) — синтаксис такой же, как у `БазовыйТип[N]` в объявлении переменной; правила допустимости базового типа массива те же, что в §3.2.

При инициализации структуры через литерал (`Точка { 1, 2 }`) обязателен позиционный порядок значений, совпадающий с порядком объявления полей, количество значений обязано точно совпадать с количеством полей, и **каждое значение проверяется на `types_compatible` с типом соответствующего поля** — в том числе если поле является массивом (в этом случае сравнение массивных типов учитывает канонизацию, см. §3.2).

## 4. Явное приведение типов (CastExpr)
Синтаксис приведения: `как <Тип>(Выражение)`.

На уровне семантического анализатора допустимы только целевой и исходный тип из набора `{целое, вещ, лог}` (`is_castable_type`); в остальном операция приведения типа выполняет принудительную подмену возвращаемого типа выражения на `Тип` без проверки совместимости с исходным типом выражения (её и не может быть — весь смысл приведения в том, чтобы разрешить менять тип, не совместимый напрямую).

В рантайме реализованы все 9 комбинаций `{целое, вещ, лог} → {целое, вещ, лог}`: числовые преобразования `целое ↔ вещ` через `static_cast` (с усечением при `вещ → целое`), а также `лог → целое`/`вещ` (`истина`→`1`/`1.0`, `ложь`→`0`/`0.0`) и `целое`/`вещ` `→ лог` (любое ненулевое значение → `истина`, ноль → `ложь`). Приведение между одинаковыми типами (например, `как<целое>(целое_значение)`) — тождественно.

## 5. Правила проверок в выражениях

* **Присваивание (`=`)**: тип левой части (L-Value) обязан быть `types_compatible` с типом правой части (R-Value) — это относится как к простой переменной, так и к элементу массива, полю структуры или переменной-массиву целиком.
* **Бинарные операторы**:
  * Операнды всегда должны иметь одинаковый тип (с учётом алиасов и канонизации массивов, см. §3.2). Нельзя, например, сложить `целое` и `вещ`.
  * **Операторы отношения** (`<`, `>`, `<=`, `>=`): допускается их применение **только** к типам `целое` и `вещ`. Сравнение `символ`, беззнаковых чисел (`бцелое`), строк, логических значений, массивов и структур вызывает статическую ошибку. Возвращают тип `лог`.
  * **Операторы равенства** (`==`, `!=`): применимы ко всем типам при условии строгого совпадения типов левого и правого операндов. Возвращают `лог`. В рантайме это сравнение **по значению**: посимвольное для строк, поэлементное/попольное (рекурсивно) для массивов и структур — а не сравнение адреса блока памяти (см. `SEMANTICS.md` §5.1).
  * **Арифметические операторы** (`+`, `-`, `*`, `/`, `%`): разрешены только для совпадающих типов (`%` — только для `целое`). Возвращают тип операндов.
* **Встроенные функции** (`печать`, `ввод`, `выход`, `паника`, `утверждение`): для каждой статически зафиксирована точная арность (`ввод` — 0 аргументов, остальные — ровно 1); вызов с иным количеством аргументов — статическая ошибка. Типы аргументов встроенных функций не проверяются (вычисляются, но не сравниваются с ожидаемым типом).

## 6. Внутреннее устройство (`TypeTable`)

Во время работы семантического анализатора все встреченные типы регистрируются в глобальной таблице `TypeTable`. Каждый тип имеет целочисленный идентификатор `TypeId`.
Использование глобальной таблицы типов и числовых идентификаторов избавляет интерпретатор от необходимости глубокого сравнения строк и структур при тайп-чекинге, делая проверку типов `O(1)`.

Структуры и алиасы естественно уникальны по имени (`TypeTable::find_by_name`/`register_type` с `name_id`), поэтому повторное упоминание одного и того же имени всегда возвращает один и тот же `TypeId`. У типа массива имени нет, поэтому для него отдельно поддерживается кэш канонизации `Analyzer::array_type_cache: (базовый TypeId, размер) → TypeId` (см. §3.2) — без него структурно идентичные, но отдельно зарегистрированные массивы были бы неотличимы от разных типов при прямом сравнении `TypeId`, что ломало бы как присваивание массива целиком, так и сравнение массивов оператором `==`, и сверку сигнатур функций с массивными параметрами.
