# Семантика и выполнение программы

Документ описан строго по фактическому коду `src/Semantic.cpp`, `inc/Semantic.hpp`, `src/Interpreter.cpp`, `inc/Interpreter.hpp`. Помимо номинальных правил, отдельно зафиксированы реальные особенности поведения текущей реализации (включая расхождения между статическим анализом и рантаймом), поскольку именно так ведёт себя код сейчас.

## 1. Архитектура и представление (Data-Oriented AST)

* Дерево плоское (Flat AST): узлы хранятся в непрерывном `std::vector<ASTNode>`, связи "родитель-ребёнок" — через индексы (`NodeId`) в общий `child_indices`.
* `Analyzer` (семантический анализатор) выполняет ровно один рекурсивный проход по дереву (`Analyzer::check`), для каждого узла один раз вычисляя и кэшируя его результирующий `TypeId` в `node_resolved_types`. Этот кэш (`get_node_type`) нигде не используется интерпретатором — `Interpreter.cpp` его не читает и всегда заново выводит нужную информацию из самих токенов узла (см. §8).
* Семантическая проверка и интерпретация — два полностью независимых прохода: `Analyzer` не влияет на рантайм напрямую (кроме факта, что при ошибке типов процесс уже завершится через `exit(1)` до старта `VM`).

## 2. Управление памятью (Arena Allocator)

* Рантайм использует Arena/bump-allocator: выделение — это сдвиг указателя, `O(1)`, без поштучного `free`.
* В Арене выделяются: интернированные строки (`StringPool`), содержимое литералов массивов (`ArrayLiteral` → `Value::as.arr`), содержимое литералов структур (`StructLiteral` → `Value::as.arr`), а также строки, введённые через `ввод()`.
* Массивы и структуры в `Value` хранятся как указатель (`as.arr`) на блок ячеек `Value` в Арене. Копирование `Value` (например, при передаче параметра в функцию или присваивании) копирует **только указатель**, а не данные — то есть массивы и структуры де-факто разделяют память с оригиналом (см. §6).

## 3. Области видимости и переменные (Scoping)

* `SymbolTable` (статический анализ) и `Environment` (рантайм) — независимые стеки областей видимости со схожей логикой: `enter_scope`/`exit_scope`, поиск — от внутренней области к внешней.
* Повторное объявление переменной в **одной и той же** области видимости — статическая ошибка "Повторное объявление переменной". Между вложенными областями работает затенение (shadowing).
* **Критическая особенность рантайма — глобальные переменные фактически не работают.** `VM::run` выполняет так:
  1. `execute(root)` — один проход по всему `Program`, где заодно регистрируются все функции (`functions[...] = id`) и выполняются top-level `VarDecl` (глобальные переменные). Но узел `Program` в `execute()` обрабатывается **тем же кодом, что и `Block`**: перед обходом детей вызывается `env.enter_scope()`, после — `env.exit_scope()`. Это означает, что вся область видимости, в которой были определены глобальные переменные, уничтожается сразу по завершении этого прохода.
  2. Затем `VM::run` находит функцию `Начало` и напрямую выполняет `execute(...)` на её теле (`Block`), которое пушит уже новую, отдельную область видимости поверх пустого глобального стека.
  * Итог: любая переменная, объявленная на верхнем уровне программы (вне функций), к моменту старта `Начало` уже недоступна. `Environment::get` не бросает ошибку на необъявленное имя — просто возвращает `Value()` (тип `Void`, `0`/`false`/`nullptr`) молча. Т.е. обращение к "глобальной" переменной внутри любой функции незаметно даёт нулевое/пустое значение, а не ошибку.
  * Практический вывод: все переменные, нужные в `Начало` и других функциях, следует объявлять **внутри тела соответствующей функции**, а не на верхнем уровне файла.
* Функции не образуют замыканий: при вызове функции (`Call` в `eval`) стек областей видимости рантайма полностью заменяется на `[глобальная-пустая-область, новая-локальная-область]` (`env.set_scopes(...)`), поэтому тело функции не видит локальных переменных вызывающего кода — только свои параметры/локальные объявления. Это согласуется со статическим анализом: `FuncDecl` в `Analyzer` тоже открывает новую область видимости и не наследует локальные переменные объемлющего блока (объявление функций внутри блоков запрещено грамматикой).

## 4. Типы: правила совместимости

* Никаких неявных приведений типов нет — ни в арифметике, ни в сравнениях, ни при передаче аргументов, ни при `вернуть`, ни при присваивании. Единственный способ преобразовать `целое ↔ вещ` (или иное разрешённое приведение) — оператор `как<T>(...)`.
* `types_compatible(t1, t2)`: истинно, если типы совпадают после разворачивания алиасов (`resolve_alias`), либо если оба — массивы с одинаковым размером и структурно совместимым базовым типом. Используется при: инициализации переменной (`ВещДекл`), передаче аргументов в функцию, инициализации полей структурного литерала, элементов литерала массива, `вернуть`.
* Инструкция присваивания (`AssignStmt`, т.е. `x = expr;` **после** объявления, не инициализация) сравнивает типы **не** через `types_compatible`, а через сырое равенство `resolve_alias(target_type) == resolve_alias(val_type)`.
  * Для скалярных типов (целое/вещ/лог/символ/строка) и для структур разницы нет — они дают один и тот же `TypeId` при каждом обращении, всё работает штатно.
  * **Для массивов это критично отличается**: каждый раз при проверке узла `ArrayLiteral` (`Analyzer::check`) регистрируется **новый** `TypeId` массива (`type_table.register_type(...)`), даже если по структуре (размер + базовый тип) он идентичен ранее объявленному. Поэтому повторное присваивание целого массива новым литералом после его объявления —
    ```
    целое[3] a = [1, 2, 3];
    a = [4, 5, 6];   // статическая ошибка "Несоответствие типов при присваивании"
    ```
    всегда падает со статической ошибкой, хотя по смыслу это корректный код. Присваивание **элементов** массива (`a[i] = x;`) и **полей** структуры (`s.f = x;`) этой проблеме не подвержено, так как там сравниваются типы элемента/поля (скаляр или структура), а не сам тип массива.

## 5. Операторы

* Перед проверкой конкретного оператора `check_binary` требует, чтобы `resolve_alias(left) == resolve_alias(right)` — иначе статическая ошибка "Несоответствие типов в бинарной операции" (это отсекает, например, смешение `целое` и `вещ` в одном выражении без явного `как<>`).
* `&&`, `||` — только между `лог`, результат `лог`.
* `<`, `<=`, `>`, `>=` — только между `целое`/`целое` или `вещ`/`вещ`, результат `лог`.
* `==`, `!=` — разрешены для любой пары **одинаковых** (после разворачивания алиасов) типов, включая массивы, структуры, строки, символы — без дополнительных ограничений по `kind`. Результат — `лог`.
* `+`, `-`, `*` — только `целое`/`целое` или `вещ`/`вещ`, результат — тип операндов.
* `/` — то же, плюс рантайм-проверка деления на ноль (`Interpreter::panic("Деление на ноль")`).
* `%` — только `целое`/`целое`.
* Унарный `-` — только к `целое`/`вещ`. Унарный `!` — только к `лог`. Унарного `+` не существует (см. `GRAMMAR.md`).

### 5.1 Особенности рантайм-вычисления операторов (реальный код `Interpreter::eval`)

* `Value` — это `union` (`i64`/`f64`/`b`/`s`/`arr`), интерпретатор при вычислении бинарных операций **не всегда** обращается к нужному члену union'а:
  * `+`, `-`, `*` — корректно ветвятся по `left.kind` (`Int` → `.i64`, иначе → `.f64`), тут проблем нет.
  * `==`/`!=` и все реляционные (`<`,`<=`,`>`,`>=`) реализованы через сравнение **`.as.i64`** независимо от фактического типа операндов (в коде помечено комментарием "Упрощенно").
    * Для `целое` это корректно всегда.
    * Для `вещ`: побитовое переинтерпретирование `double` как `int64` **случайно** сохраняет порядок сравнения, если оба операнда положительные (или один операнд отрицательный, а другой положительный — тогда сравнение тоже корректно, поскольку у отрицательных float старший бит установлен и итоговое `int64`-значение всегда меньше любого положительного). Но сравнение **двух отрицательных** вещественных чисел друг с другом (`-2.0 < -1.0` и т.п.) даёт **неверный результат** — порядок инвертируется. Для `==`/`!=` с `вещ` результат корректен (побитовое совпадение бит эквивалентно равенству значений, за исключением редких краевых случаев `-0.0`/`NaN`, не встречающихся в типичных примерах).
    * Для `строка`: `==`/`!=` фактически сравнивает указатели (`as.s`, переинтерпретированные как `i64`), т.е. это ссылочное, а не посимвольное сравнение. На практике совпадает с ожидаемым результатом для строковых **литералов**, потому что `StringPool` интернирует (дедуплицирует) одинаковые по содержимому строки, и одинаковый текст всегда получает один и тот же указатель в памяти. Для строк, полученных через `ввод()` (не интернируются, каждый вызов — новое выделение в Арене), сравнение по содержимому работать не будет даже при одинаковом тексте.
    * Для массивов/структур `==`/`!=` статически разрешены (типы совпадают), но по факту сравнивают адрес блока в Арене (`as.arr` как `i64`) — то есть это сравнение "один и тот же ли это объект", а не поэлементное сравнение значений.
  * Проверка деления на ноль (`right.as.i64 == 0 && right.as.f64 == 0.0`) работает корректно для обоих типов, поскольку битовое представление нуля совпадает у `int64` и `double` (все биты равны 0), поэтому это не является багом, несмотря на использование union-полей "не по назначению".

## 6. Функции и вызовы

* Функция определяется уникально по полному имени (`ПространствоИмён::Имя` либо просто `Имя` на верхнем уровне). Повторное объявление с **другой** сигнатурой (иным типом возврата или параметров) — статическая ошибка, даже если это по сути другая перегрузка: **перегрузка функций не поддерживается**.
* Разрешён ровно один прототип (`функция ф(...): T;`) перед одним определением (`функция ф(...): T { ... }`); повторное определение уже определённой функции — статическая ошибка.
* Аргументы проверяются на точное количество и `types_compatible`-совместимость по типам (без неявных преобразований).
* `вернуть Expr;` — тип `Expr` обязан быть `types_compatible` с объявленным типом возврата. Если функция объявлена как `пусто`, допустим голый `вернуть;`.
* Проверка "все пути возврата гарантируют значение" (`all_paths_return`) — для функций с ненулевым типом возврата: учитывает `Return`, рекурсивно — оба ветки `Block`/`If` (только если есть `иначе`), а также вызовы встроенных `паника`/`выход` (они считаются гарантированным завершением потока). `Continue`/`While`/произвольные вызовы других функций не считаются "гарантирующими возврат" — например, тело `пока(истина) { вернуть 1; }` не пройдёт эту проверку, так как `While` не входит в анализ `all_paths_return` вообще (всегда даёт `false` по default-ветке `switch`), даже если цикл бесконечен и содержит `вернуть` внутри.
* **Отсутствие точки входа `Начало` не является статической ошибкой.** `Analyzer` вообще не проверяет наличие такой функции. Отсутствие обнаруживается только в `VM::run` в момент запуска рантайма и приводит к **runtime-паника** ("Точка входа 'Начало' не найдена", формат "Ошибка времени выполнения"), а не к "статической ошибке" в духе остальных проверок парсера/анализатора.
* Аргументы передаются по значению (копируется `Value`), но для массивов и структур `Value` — это указатель на общий блок памяти в Арене, поэтому **мутации элементов/полей массива или структуры, переданных в функцию, видны вызывающей стороне** (де-факто передача по ссылке для составных типов).
* Рекурсия ограничена глубиной 1000 вызовов (`call_stack.size() > 1000`) — превышение даёт runtime-панику "Превышен лимит глубины рекурсии (Stack Overflow)".
* При панике (`VM::panic`) печатается текущая строка и стек вызовов (`call_stack`), затем `exit(1)`.

## 7. Встроенные функции (Built-ins)

Функции `печать`, `ввод`, `выход`, `паника`, `утверждение` не имеют формальной сигнатуры в `Analyzer`: разрешено передавать **произвольное количество аргументов любых типов** — они лишь вычисляются (`check(...)`), но не проверяются по типу/количеству.

Однако в `Interpreter::eval` реализация этих функций **предполагает наличие как минимум одного аргумента** и не проверяет размер `args`:
* `печать(...)` — печатает **только `args[0]`** (целое/вещ/лог/строка/символ с переводом строки); остальные аргументы вычисляются (их побочные эффекты происходят), но не выводятся. Если аргументов 0 — обращение к `args[0]` является неопределённым поведением (реальный краш/непредсказуемый результат, т.к. `std::vector::operator[]` не проверяет границы).
* `ввод(...)` — любые переданные аргументы полностью игнорируются; считывает строку из `stdin`, возвращает `строка`, память под неё выделяется в Арене (без интернирования — см. §5.1 про сравнение строк).
* `выход(...)` — вызывает `std::exit(args[0].as.i64)`; вызов без аргументов — неопределённое поведение (обращение к несуществующему `args[0]`).
* `паника(...)` — печатает `args[0].as.s` как сообщение об ошибке и завершает процесс; тот же риск при отсутствии аргументов.
* `утверждение(...)` — проверяет `args[0].as.b`, при `ложь` — паника "Утверждение ложно"; тот же риск при отсутствии аргументов.

Практический вывод: на этих встроенных функциях всегда нужно передавать **ровно один** (значимый) аргумент, хотя статически это не требуется и не проверяется.

## 8. Приведение типов (`как<T>(...)`)

* Статически разрешено приведение только между `T ∈ {целое, вещ, лог}` (`is_castable_type`), причём **оба** типа — и целевой, и исходный — обязаны входить в этот набор. Приведение к/из `символ`, `строка`, структур, массивов, `пусто` — статическая ошибка.
* Тип результата выражения `как<T>(...)` для статического анализа — это ровно `T` (без какой-либо проверки соответствия рантайм-значения).
* **В рантайме (`Interpreter::eval`, узел `Cast`) реально реализованы только приведения к `целое` и `вещ`** (конвертация `Int↔Float` через `static_cast`). Приведение к `лог` статически разрешено, но в рантайме падает в ветку `else { res = expr_val; }` — то есть **не выполняет никакого реального преобразования**: результат — это исходное значение as-is (с исходным `kind`, исходным битовым представлением). Из-за совпадения представления "0 = ложь / не-0 = истина" в младшем байте union'а, `как<лог>(0)` и `как<лог>(1)` на практике часто ведут себя правдоподобно в булевых контекстах, но это побочный эффект компоновки памяти, а не осознанная семантика приведения.

## 9. Массивы

* Базовым типом массива статически разрешены: `целое`, `вещ`, `лог`, `символ`, `строка`, а также **структуры** (`is_allowed_array_base_type` включает `TypeKind::Struct`) — несмотря на то, что тексты соответствующих статических ошибок говорят "только примитив или строка". Массив массивов запрещён (многомерные массивы) — проверяется отдельно на уровне парсера (см. `GRAMMAR.md`).
* Литерал массива не может быть пустым (`[]` как значение — статическая ошибка "Пустые массивы не поддерживаются"); тип базового элемента определяется по первому элементу, остальные элементы должны быть `types_compatible` с ним.
* Индексация (`a[i]`) требует индекса типа `целое`, база должна быть типом `Array`; результат — тип элемента.
* **Границы индекса не проверяются ни статически, ни в рантайме** — `arr_val.as.arr[idx_val.as.i64]` выполняется напрямую; выход за пределы — неопределённое поведение (в типичном случае сегфолт или порча памяти Арены).
* Присваивание элементу массива (`a[i] = x;`) типобезопасно и работает штатно (сравниваются типы элемента, а не массива — см. §4). Присваивание **всего массива целиком** после его объявления — статическая ошибка (см. §4, критическая особенность).

## 10. Структуры

* `структура` должна содержать хотя бы одно поле (пустая структура — статическая ошибка). Прямая рекурсия поля на свой же тип структуры запрещена (`Рекурсивное объявление структуры`); проверка простая (сравнение с `struct_id` текущей структуры), взаимная/косвенная рекурсия между двумя структурами не проверяется явно.
* Поля хранятся как последовательные "слоты" (`offset`), каждое поле занимает ровно 1 слот независимо от базового типа — структуры физически представлены как плоский массив `Value` (`Value::as.arr`) в Арене.
* Литерал структуры (`Имя{ ... }`) — обязателен **позиционный** порядок значений, совпадающий с порядком объявления полей; количество значений обязано точно совпадать с количеством полей; каждое значение проверяется на `types_compatible` с типом соответствующего поля.
* Доступ к полю (`s.f`) статически резолвится в числовой `offset`, который записывается прямо в `extra_data` узла `MemberAccess`; в рантайме — просто `obj_val.as.arr[offset]`, без какой-либо проверки, что `obj_val` реально указывает на структуру этого типа (доверие к результату статического анализа).
* Присваивание в поле (`s.f = x;`) типобезопасно (сравниваются типы поля, а не структуры — проблема §4 сюда не относится, так как структура всегда получает один и тот же `TypeId`, зарегистрированный один раз при объявлении `структура`, в отличие от массивов).

## 11. Управление потоком

* `если(...)`/`иначе` — условие обязано быть строго `лог` (никакого приведения из `целое`, в отличие от C). В рантайме `cond.as.b` читается напрямую (учитывая, что статический анализ уже гарантировал тип `лог`, риска чтения "не того" union-поля здесь нет).
* `пока(...)` — то же условие на тип; `прервать`/`продолжить` — статически проверяется, что они находятся внутри цикла (`loop_depth`); в рантайме реализованы через флаги `should_break`/`should_continue`, проверяемые после каждого `execute` внутри тела цикла и внутри `Block`/`Program`.
* `вернуть` без выражения возможен только если функция объявлена как `пусто`.

## 12. Пространства имён

* `пространство Имя { ... }` реализовано как префикс имени функции (`Имя::`) в единой плоской таблице функций — как в `Analyzer`, так и в `VM`, независимо друг от друга поддерживающих свой `namespace_prefix`. Вложенные пространства имён накапливают префикс (`A::B::имя`).
* Точка входа `Начало` ищется по буквальному имени `"Начало"` без какого-либо префикса — если объявить `Начало` внутри пространства имён, `VM::run` её не найдёт (см. §6).

## 13. Итоговый список известных особенностей реализации (сводка)

1. Глобальные переменные верхнего уровня фактически недоступны внутри функций (в т.ч. `Начало`) — область видимости, где они инициализируются, закрывается до старта `Начало` (§3).
2. Присваивание (не инициализация) переменной-массива новым литералом массива всегда даёт статическую ошибку "Несоответствие типов при присваивании", хотя семантически корректно (§4).
3. Сравнение отрицательных `вещ` чисел друг с другом через `<`, `<=`, `>`, `>=` может давать неверный результат из-за побитового сравнения через `.i64` (§5.1).
4. `==`/`!=` для строк — это сравнение указателей, а не содержимого (совпадает с ожидаемым только благодаря интернированию литералов; для строк из `ввод()` не совпадает). Для массивов/структур `==`/`!=` — тоже сравнение адреса блока, а не значений (§5.1).
5. Встроенные функции `печать`/`выход`/`паника`/`утверждение`, вызванные без аргументов, приводят к неопределённому поведению в рантайме, несмотря на то, что статический анализ такой вызов пропускает (§7). `печать` с несколькими аргументами выводит только первый.
6. Отсутствие точки входа `Начало` обнаруживается только в рантайме как паника, а не как статическая ошибка (§6).
7. Приведение `как<лог>(...)` статически разрешено, но фактически не реализовано в рантайме (нет реального преобразования значения) (§8).
8. Массивы структур разрешены статическим анализом, хотя тексты ошибок валидации утверждают обратное (§9).
9. Индексация массива не проверяет границы ни на одном из этапов (§9).
10. Проверка "все пути возврата дают значение" не учитывает `While`/`Continue` — тело функции с единственным `вернуть` внутри `пока(истина){...}` не проходит эту проверку, даже если цикл гарантированно вернёт значение (§6).
