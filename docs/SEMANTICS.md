# Семантика и выполнение программы

Помимо номинальных правил, отдельно зафиксированы реальные особенности поведения текущей реализации.

## 1. Архитектура и представление (Data-Oriented AST)

* Дерево является плоским (Flat AST): узлы хранятся в виде последовательного списка, а связи "родитель-ребенок" осуществляются через индексы в общем списке дочерних элементов.
* Семантический анализатор выполняет ровно один рекурсивный проход по дереву, вычисляя и кэшируя для каждого узла его результирующий тип. Этот кэш типов не используется интерпретатором, который при выполнении выводит необходимую информацию заново из токенов узлов (см. параграф 8).
* Семантическая проверка и интерпретация - два полностью независимых прохода. Результаты статического анализа не влияют на выполнение напрямую (кроме случая, когда при наличии статической ошибки выполнение программы прерывается до запуска интерпретатора). Интерпретатор доверяет гарантиям статического анализа (например, количеству и типам параметров функций, наличию точки входа) и не дублирует эти проверки (см. параграф 6, параграф 7).

## 2. Управление памятью

* Память для динамических данных (таких как строки, массивы и структуры) выделяется в общей области динамической памяти.
* Массивы и структуры передаются и присваиваются по ссылке (копируется только ссылка на область памяти и количество элементов/полей), а не по значению. Сами данные при этом не дублируются, поэтому массивы и структуры разделяют память с оригиналом (см. параграф 6).
* Количество элементов массива или полей структуры фиксируется при их создании и используется для проверки границ индекса (см. параграф 9) и для глубокого сравнения значений (см. параграф 5.1).

## 3. Области видимости и переменные (Scoping)

* Статический анализ и выполнение используют концепцию вложенных областей видимости: поиск переменных идет от внутренней области к внешней.
* Повторное объявление переменной в одной и той же области видимости - статическая ошибка "Повторное объявление переменной". Между вложенными областями работает затенение (shadowing).
* **Глобальные переменные верхнего уровня.** Переменные, объявленные на верхнем уровне программы (вне функций), попадают в глобальную область видимости, которая живет до завершения работы программы.
  * Итог: любая глобальная переменная доступна на чтение и на запись из любой функции, включая `Начало`.
* Функции не образуют замыканий: при вызове функции тело функции не видит локальных переменных вызывающего кода - доступны только ее параметры, локальные переменные и глобальные переменные. Объявление функций внутри блоков запрещено правилами языка.
  * Изменения глобальных переменных внутри функций сохраняются и видны во всех последующих обращениях к ним, независимо от глубины вложенности вызовов.

## 4. Типы: правила совместимости

* Никаких неявных приведений типов нет — ни в арифметике, ни в сравнениях, ни при передаче аргументов, ни при `вернуть`, ни при присваивании. Единственный способ преобразовать `целое ↔ вещ ↔ лог` — оператор `как<T>(...)` (см. §8).
* `types_compatible(t1, t2)`: истинно, если типы совпадают после разворачивания алиасов (`resolve_alias`), либо если оба — массивы с одинаковым размером и структурно совместимым базовым типом. Используется при: инициализации переменной (`ВещДекл`), присваивании (`AssignStmt`), передаче аргументов в функцию, инициализации полей структурного литерала, элементов литерала массива, `вернуть`.
* **Канонизация типов массивов.** Тип массива — не именованный (в отличие от структуры), поэтому каждое упоминание вида `БазовыйТип[N]` (объявление переменной, поле структуры, параметр/возврат функции, синоним типа) или литерал `[a, b, c]` в принципе могло бы регистрировать собственный, отдельный `TypeId`, даже если по структуре (размер + базовый тип) он идентичен уже встречавшемуся. Чтобы этого не происходило, регистрация типа массива всегда проходит через `Analyzer::canonical_array_type(base_type, size)`, которая хранит кэш `array_type_cache: (raскрученный_базовый_тип, размер) → TypeId` и возвращает уже существующий `TypeId`, если такой массив уже был зарегистрирован — новый тип создаётся только при первой встрече конкретной формы `(базовый тип, размер)`. Все места, порождающие тип массива (`VarDecl`, поле `StructDecl`, параметр/возврат `FuncDecl`, `TypeAlias`, `ArrayLiteral`), проходят либо непосредственно через `canonical_array_type`, либо через обёртку `wrap_array_if_needed` (которая дополнительно проверяет допустимость базового типа — см. §9 — и делегирует регистрацию туда же).
  * Следствие: два массива с одинаковым базовым типом и размером, объявленные в разных местах программы, получают **один и тот же** `TypeId`. Это делает корректными не только `types_compatible`, но и прямые сравнения `TypeId == TypeId`, используемые, например, в `check_binary` (для `==`, `<` и т.п.) и при сверке сигнатур прототипа/определения функции — благодаря канонизации оба места теперь корректно считают структурно одинаковые массивы одним типом без необходимости отдельно приводить их через `types_compatible`.
  * Присваивание переменной-массиву нового литерала массива (`целое[3] a = [1,2,3]; a = [4,5,6];`) теперь корректно проходит статическую проверку: `AssignStmt` сравнивает типы через `types_compatible` (а не через сырое равенство `TypeId`, как было раньше), и благодаря канонизации `final_type` переменной и `result` литерала — это в любом случае один и тот же `TypeId`. Присваивание с **другим размером** (`a = [1, 2];`) остаётся статической ошибкой "Несоответствие типов при присваивании", как и должно быть.

## 5. Операторы

* Бинарные операторы требуют совпадения типов операндов (после раскрытия синонимов), иначе возникает статическая ошибка "Несоответствие типов в бинарной операции" (например, нельзя смешивать `целое` и `вещ` без явного приведения). Благодаря канонизации типов массивов (см. параграф 4) это правило корректно выполняется и для двух структурно одинаковых массивов.
* `&&`, `||` - только между `лог`, результат `лог`.
* `<`, `<=`, `>`, `>=` - только между `целое`/`целое` или `вещ`/`вещ`, результат `лог`.
* `==`, `!=` - разрешены для любой пары одинаковых (после раскрытия синонимов) типов, включая массивы, структуры, строки, символы. Результат - `лог`. В процессе выполнения это сравнение по значению (см. параграф 5.1).
* `+`, `-`, `*` - только `целое`/`целое` или `вещ`/`вещ`, результат - тип операндов.
* `/` - то же, плюс проверка деления на ноль во время выполнения с генерацией ошибки.
* `%` - только `целое`/`целое`, проверка деления на ноль во время выполнения применяется и здесь.
* Унарный `-` - только к `целое`/`вещ`. Унарный `!` - только к `лог`. Унарного `+` не существует (см. `GRAMMAR.md`).

### 5.1 Вычисление операторов во время выполнения

* Бинарные операции выполняются в соответствии с типами их операндов:
  * `+`, `-`, `*` - выполняются как целочисленные или вещественные операции.
  * `%` - выполняется только для целых чисел, с проверкой деления на ноль.
  * `<`, `<=`, `>`, `>=` - выполняются как сравнения соответствующих типов (целых или вещественных). Сравнение вещественных чисел происходит корректно для любых знаков.
  * `==`, `!=` - реализуют сравнение по значению:
    * `целое` / `символ` / `лог` - прямое сравнение значений.
    * `вещ` - стандартное сравнение чисел с плавающей точкой.
    * `строка` - сравнение по фактическому содержимому строк, а не по их адресу в памяти.
    * `массив` / `структура` - поэлементное/попольное (рекурсивное) сравнение значений. Сначала проверяется количество элементов/полей, затем рекурсивно сравнивается каждая пара элементов или полей. Это работает для любой вложенности (например, массивов структур).
  * Деление на ноль (как для `/`, так и для `%`) проверяется во время выполнения и вызывает ошибку.
* Унарные операторы вычисляются по токену оператора: `-` меняет знак числа (целого или вещественного), `!` инвертирует логическое значение.

## 6. Функции и вызовы

* Функция определяется уникально по полному имени (`ПространствоИмён::Имя` либо просто `Имя` на верхнем уровне). Повторное объявление с другой сигнатурой (иным типом возврата или параметров) - статическая ошибка. Перегрузка функций не поддерживается. Сравнение сигнатур прототипа и определения учитывает канонизацию типов массивов (см. параграф 4).
* Разрешён ровно один прототип (`функция ф(...): T;`) перед одним определением (`функция ф(...): T { ... }`); повторное определение уже определённой функции - статическая ошибка.
* Количество аргументов при вызове должно точно соответствовать количеству параметров функции, а их типы должны быть совместимы (без неявных приведений). Параметры и возвращаемое значение могут быть массивами (см. параграф 9).
* При возврате значения (`вернуть выражение;`) тип выражения должен быть совместим с объявленным типом возврата. Если функция объявлена с типом `пусто`, допускается пустой оператор `вернуть;`.
* Проверка возврата значения из всех ветвей учитывает операторы `вернуть`, рекурсивный обход ветвления `если/иначе`, вызовы встроенных функций аварийного завершения (`паника`, `выход`), а также циклы `пока`, если условие цикла гарантированно истинно и в его теле нет прерываний.
* **Отсутствие точки входа `Начало` - статическая ошибка.** Функция с именем `"Начало"` (на верхнем уровне, без префикса пространства имен) должна быть обязательно определена, иначе программа не запустится.
* Аргументы передаются по значению. Однако для массивов и структур копируется только ссылка на область памяти, поэтому изменения элементов массива или полей структуры внутри функции будут видны вызывающему коду.
* Глубина рекурсии ограничена пределом в 1000 вызовов. Превышение лимита вызывает ошибку времени выполнения ("Stack Overflow").

## 7. Встроенные функции (Built-ins)

Для встроенных функций (`печать`, `ввод`, `выход`, `паника`, `утверждение`) статически проверяется точное количество аргументов: `ввод` требует 0 аргументов, остальные - ровно 1. Вызов с иным количеством аргументов вызывает статическую ошибку.

* `печать(x)` - выводит ровно один аргумент (целое/вещ/лог/строка/символ) на экран с переводом строки.
* `ввод()` - не принимает аргументов, считывает строку из стандартного потока ввода и возвращает ее в виде значения типа `строка`.
* `выход(код)` - завершает выполнение программы с возвратом указанного числового кода.
* `паника(сообщение)` - выводит сообщение об ошибке и аварийно завершает процесс.
* `утверждение(условие)` - проверяет логическое условие: если оно ложно, программа аварийно завершается с сообщением о ложности утверждения.

Типы аргументов встроенных функций статически не проверяются, кроме соответствия по количеству параметров. Ответственность за передачу корректных типов (например, целого числа в `выход`) лежит на авторе кода.

## 8. Приведение типов (`как <Тип>(...)`)

* Статически разрешено явное приведение только между типами из набора `{целое, вещ, лог}`. Приведение к другим типам или из них вызывает статическую ошибку.
* Типом результата выражения приведения для статического анализа становится целевой тип.
* Во время выполнения поддерживаются все комбинации преобразований между `{целое, вещ, лог}`:
  * Приведение к `целое`: вещественные числа усекаются, логические преобразуются в `1` (истина) или `0` (ложь).
  * Приведение к `вещ`: целые числа переводятся в вещественные, логические преобразуются в `1.0` или `0.0`.
  * Приведение к `лог`: любое ненулевое число (как целое, так и вещественное) превращается в `истина`, нулевое значение - в `ложь`.

## 9. Массивы

* Базовым типом массива могут быть: `целое`, `вещ`, `лог`, `символ`, `строка`, а также структуры. Многомерные массивы (массивы массивов) запрещены как на уровне синтаксиса, так и семантически.
* Литерал массива не может быть пустым; тип элементов определяется по первому элементу, а остальные должны быть совместимы с ним.
* **Массив как часть типа в любом контексте.** Объявление массива `БазовыйТип[N]` допускается в объявлении переменной, в качестве поля структуры (`структура Т { целое[3] поле; }`), параметра или возвращаемого значения функции, а также как цель синонима типа (`тип Тройка = целое[3];`). Во всех случаях базовый тип массива должен быть допустимым. Допустима как структура, содержащая поле-массив, так и массив структур, и их комбинации, если один массив не вложен непосредственно в другой.
  * Массив-поле структуры того же типа, что и сама структура (например, `структура Узел { Узел[2] дети; }`), разрешен и не считается запрещенным рекурсивным объявлением.
* Индексация (`a[i]`) требует индекса типа `целое`; результатом выражения является тип элемента массива.
* **Проверка границ индекса во время выполнения.** При обращении к элементу массива по индексу (как для чтения, так и для записи) проверяются границы массива. Если индекс отрицательный или выходит за границы размера массива, программа аварийно завершается с ошибкой ("Индекс выходит за границы массива").
* Присваивание элементу массива (`a[i] = x;`) проверяет соответствие типов элементов. Присваивание всего массива целиком новым литералом того же размера также корректно.

## 10. Структуры

* Структура должна содержать хотя бы одно поле (пустая структура - статическая ошибка). Прямая рекурсия поля на свой же тип структуры по значению запрещена, но рекурсия через поле-массив того же типа разрешена (см. параграф 9).
* Литерал структуры (`Имя{ ... }`) требует строго позиционного порядка значений, совпадающего с порядком объявления полей. Количество переданных значений должно точно совпадать с количеством полей, и каждое значение должно быть совместимо с типом соответствующего поля.
* Доступ к полю структуры (`s.f`) разрешается статически. При выполнении значение поля извлекается без дополнительных динамических проверок типов, основываясь на результатах статического анализа.
* Присваивание значения полю структуры (`s.f = x;`) типобезопасно и проверяет совместимость типа присваиваемого выражения с объявленным типом этого поля.

## 11. Управление потоком

* `если (...)` / `иначе` - условие ветвления должно иметь строго логический тип `лог`. Во время выполнения условие оценивается напрямую без неявных приведений.
* `пока (...)` - условие цикла должно иметь строго логический тип `лог`. Статический анализ проверяет, что операторы `прервать` и `продолжить` вызываются исключительно внутри циклов.
* Оператор `вернуть` без выражения допускается только в функциях, возвращающих тип `пусто`.

## 12. Пространства имён

* `пространство Имя { ... }` позволяет группировать функции с префиксом пространства имен (`Имя::`). Вложенные пространства имён накапливают префикс (`A::B::имя`).
* Точка входа `Начало` ищется по буквальному имени `"Начало"` без какого-либо префикса. Если объявить `Начало` внутри пространства имён, программа не распознает её как точку входа, и статический анализ выдаст ошибку "Точка входа 'Начало' не найдена" (см. параграф 6).
