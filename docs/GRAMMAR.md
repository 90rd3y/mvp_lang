# Грамматика Языка

Документ описывает лексическую и синтаксическую грамматику в формате EBNF.
Грамматика построена строго по фактическому поведению `src/Lexer.cpp`, `inc/Lexer.hpp`,
`src/Parser.cpp`, `inc/Parser.hpp` — то есть описывает язык, который **реально принимает**
текущий парсер, включая его ограничения, дыры и несогласованности. Места, требующие
доработки/решения, отмечены как **⚠ ТРЕБУЕТ РАЗРЕШЕНИЯ**.

---

## 1. Лексическая структура (Lexer)

### 1.1 Ключевые слова

Список ключевых слов взят из `Scanner::check_keyword` (единственный источник истины).
Других ключевых слов не существует.

```
целое  вещ  лог  символ  строка  пусто  пост
если  иначе  пока
прервать  продолжить  вернуть
истина  ложь
структура  тип  массив  функция  пространство  как
```

⚠ **ТРЕБУЕТ РАЗРЕШЕНИЯ**: `массив` (`KwArray`) сканируется, но нигде не используется в
`Parser.cpp`. Синтаксис массивов задаётся исключительно через `Тип[Размер] Имя`, слово
`массив` фактически мёртвое.

### 1.2 Идентификаторы

Идентификатор — это последовательность символов, где первый символ проверяется как
`isalpha(c) || (unsigned char)c > 127 || c == '_'`, а последующие —
`isalnum(c) || (unsigned char)c > 127 || c == '_'`. Условие `> 127` пропускает как
кириллицу, так и произвольные не-ASCII байты (в том числе внутри многобайтовых UTF-8
последовательностей), то есть контроль корректности UTF-8 на уровне лексера отсутствует.

```ebnf
IdentStart  ::= ASCII_LETTER | "_" | NON_ASCII_BYTE
IdentChar   ::= ASCII_LETTER | ASCII_DIGIT | "_" | NON_ASCII_BYTE
Identifier  ::= IdentStart IdentChar*
```

Если распознанная лексема совпадает с одним из ключевых слов из 1.1 — токен
переклассифицируется в соответствующее ключевое слово, иначе это `Identifier`.

Язык регистрозависим (case-sensitive), сравнение ключевых слов побайтовое.

### 1.3 Литералы

```ebnf
IntLiteral    ::= digit+
FloatLiteral  ::= digit+ "." digit+
```
Число распознаётся как `Float` только если после `.` следует цифра (`peek_next()`
проверяется через `isdigit`); иначе точка не поглощается числом (например, `5.` не
образует `Float` — `.` останется отдельным токеном `Dot`).
`Int`/`Float` не поддерживают экспоненциальную запись, знак числа, разделители разрядов
или шестнадцатеричный/восьмеричный/двоичный форматы.

```ebnf
StringLiteral ::= '"' StringChar* '"'
StringChar    ::= any-char-except-quote | "\" any-char
```
Экранирование `\` в `scan_string` лишь пропускает следующий символ без какой-либо
интерпретации (то есть `\n` в строке — это два "сырых" символа `\` и `n`, а не перевод
строки). Экранированная кавычка `\"` корректно не завершает строку.

⚠ **ТРЕБУЕТ РАЗРЕШЕНИЯ**: `TokenType::Char` объявлен в `Lexer.hpp`, но `Scanner`
**никогда не порождает** этот токен — нет сканирования символьных литералов в одинарных
кавычках. Тип `символ` можно объявить как тип данных, но литерала для него не
существует на уровне лексера.

```ebnf
BoolLiteral ::= "истина" | "ложь"
```
Это ключевые слова (`KwTrue` / `KwFalse`), а не отдельная лексическая категория.

### 1.4 Операторы и пунктуация

| Лексема | Токен | Примечание |
|---|---|---|
| `+` | Plus | |
| `-` | Minus | |
| `*` | Star | |
| `/` | Slash | если далее `/` или `*` — это комментарий, не токен |
| `%` | Percent | сканируется, но не используется в грамматике выражений (см. 2.6) |
| `=` | Equal | |
| `==` | EqualEqual | |
| `!` | Bang | сканируется, но не используется в грамматике выражений (см. 2.6) |
| `!=` | BangEqual | |
| `<` | Less | |
| `<=` | LessEqual | |
| `>` | Greater | |
| `>=` | GreaterEqual | |
| `&&` | AmpAmp | сканируется, но не используется в грамматике выражений (см. 2.6) |
| `\|\|` | PipePipe | сканируется, но не используется в грамматике выражений (см. 2.6) |
| `.` | Dot | |
| `,` | Comma | |
| `:` | Colon | |
| `::` | ColonColon | |
| `;` | Semicolon | |
| `(` `)` `{` `}` `[` `]` | LParen/RParen/LBrace/RBrace/LBracket/RBracket | |

⚠ **ТРЕБУЕТ РАЗРЕШЕНИЯ (баг лексера)**: одиночный `&`, не составляющий `&&`, обрабатывается
веткой `case '&': if (match('&')) ...` — если следующий символ не `&`, токен **не
создаётся вообще** (символ молча "проглатывается", ни ошибки, ни токена). Аналогично для
одиночного `|`. Одиночные `&` и `|` в языке не определены и приводят к тихой потере
символа во входном потоке.

### 1.5 Пробелы и комментарии

```ebnf
Whitespace   ::= " " | "\t" | "\r" | "\n"
LineComment  ::= "//" [^\n]*
BlockComment ::= "/*" ... "*/"    (* поддерживается вложенность *)
```
Блочные комментарии поддерживают вложенность (`depth` счётчик в `scan_comment`).
Пробелы и комментарии игнорируются лексером (не порождают токенов).

### 1.6 Обработка ошибок лексера

Незакрытая строка и нераспознанный символ порождают токен `TokenType::Error` с
сообщением на русском ("Незавершенная строка", "Неизвестный символ"). Дальнейшая
обработка таких токенов в парсере не предусмотрена явно — они приведут к попаданию в
общий механизм `error()` парсера (статическая ошибка) при попытке их разобрать.

## 2. Синтаксическая грамматика (Parser)

Парсер — рекурсивный спуск для деклараций/инструкций плюс Pratt-парсер для выражений.
AST — плоский, узлы хранятся в едином массиве (`nodes`), дети адресуются через
`child_indices`; узел `0` зарезервирован как `NodeType::None` (невалидный/InvalidNode).

### 2.1 Программа

```ebnf
Program ::= Decl* EOF
```

### 2.2 Тип (Type)

⚠ **ТРЕБУЕТ РАЗРЕШЕНИЯ**: правило разбора "типа" в текущем коде **не унифицировано** —
в разных контекстах тип разбирается по-разному:

1. **В объявлении переменной** (`declaration()`/`block()`): тип — это либо одно из
   ключевых слов `целое | вещ | лог | символ | строка`, либо (эвристически, по
   lookahead) `Identifier`, если после него следует ещё один `Identifier`, либо
   паттерн `Identifier [ Int ] Identifier`. Само слово-тип **потребляется одним
   токеном** — квалифицированные имена типов вида `Пространство::Тип` для
   пользовательских типов переменных **не поддерживаются** (эвристика смотрит только
   на непосредственно следующий токен).
2. **В параметрах функции, полях структуры, синониме типа, типе возврата функции**
   (`func_declaration`, `struct_declaration`, `type_alias`) тип разбирается вызовом
   `advance()` — **это буквально "любой один токен"**, без проверки, что это ключевое
   слово типа или идентификатор. Формально грамматика этих мест пропустит даже
   `+` или `;` как "тип", что будет поймано только позже (на этапе Semantic) либо
   вообще не поймано. Указывать здесь предлагается ограничение по построению (см.
   ниже), но код этого не проверяет.
3. Массивный суффикс `[Размер]` **поддерживается только для переменных** через
   `var_declaration`. Поля структур и параметры функций **не могут** быть массивами
   (в `struct_declaration`/`func_declaration` нет обработки `[`).

Формально (с учётом реальных проверок только для VarDecl, п.1):
```ebnf
BuiltinType ::= "целое" | "вещ" | "лог" | "символ" | "строка"
Type        ::= BuiltinType | Identifier        (* только для VarDecl, см. п.1 *)
TypeAny     ::= AnyToken                        (* для параметров/полей/возврата/алиаса, см. п.2, ⚠ дыра парсера *)
```

### 2.3 Объявления (Decl)

```ebnf
Decl ::= NamespaceDecl
       | FuncDecl
       | StructDecl
       | TypeAlias
       | VarDecl
       | Stmt

NamespaceDecl ::= "пространство" Identifier "{" Decl* "}"
```
Пространства имён допускают вложенность (тело — снова `Decl*`, включая другой
`NamespaceDecl`). Квалификация членов пространства имён при обращении реализована
только на уровне идентификатора-строки (`Ns::Ident`, см. 2.7 `IdentifierPath`) —
структурного разрешения имён на уровне AST парсер не производит.

```ebnf
FuncDecl ::= "функция" Identifier "(" ParamList? ")" ":" TypeAny (Block | ";")
ParamList ::= Param ("," Param)*
Param     ::= TypeAny Identifier
```
Функция без тела (оканчивающаяся на `;`) — это прототип. Список параметров без
элементов допустим (`()`).

```ebnf
StructDecl  ::= "структура" Identifier "{" Field+ "}"
Field       ::= TypeAny Identifier ";"
```
Структура обязана содержать хотя бы одно поле (иначе — статическая ошибка "Структура
должна содержать хотя бы одно поле"). Поля-массивы не поддерживаются (см. 2.2 п.3).

```ebnf
TypeAlias ::= "тип" Identifier "=" TypeAny ";"
```

```ebnf
VarDecl ::= "пост"? Type ("[" IntLiteral "]")? Identifier "=" Expr ";"
```
Ключевые аспекты:
- `пост` (const) — необязательный модификатор; при его отсутствии переменная
  мутабельна (`is_mutable = true` по умолчанию).
- Если указано `пост`, но далее не следует распознаваемый `Type` — статическая ошибка
  ("Ожидалось указание типа после ключевого слова 'пост'").
- Инициализация **строго обязательна** для всех переменных (в том числе не-`пост`) —
  `=` и выражение-инициализатор не могут быть опущены.
- Размер массива — **только целочисленный литерал** (`IntLiteral`), не произвольное
  константное выражение; при попытке указать второй `[` сразу за первым — статическая
  ошибка "Многомерные массивы запрещены" (массивы строго одномерные).
- Определение, является ли текущая конструкция объявлением переменной (в отличие от
  `Stmt`), выполняется эвристическим lookahead на 2–5 токенов вперёд (см. 2.2 п.1);
  это касается как `declaration()` (глобальный/namespace-уровень), так и `block()`
  (уровень тела блока).

### 2.4 Инструкции (Stmt)

Внутри `Block` допускаются как локальные `VarDecl`, так и любые `Stmt` (порядок
свободный, определяется тем же lookahead-эвристикой, что и в 2.3).

```ebnf
Stmt ::= IfStmt
       | WhileStmt
       | ReturnStmt
       | BreakStmt
       | ContinueStmt
       | Block
       | AssignStmt
       | ExprStmt

IfStmt       ::= "если" "(" Expr ")" Stmt ("иначе" Stmt)?
WhileStmt    ::= "пока" "(" Expr ")" Stmt
ReturnStmt   ::= "вернуть" Expr? ";"
BreakStmt    ::= "прервать" ";"
ContinueStmt ::= "продолжить" ";"
Block        ::= "{" (VarDecl | Stmt)* "}"
```
`If`/`While` принимают в качестве тела **произвольный `Stmt`**, а не обязательно
`Block` — то есть `если (x) вернуть 1;` без фигурных скобок синтаксически допустимо.

```ebnf
AssignStmt ::= Expr "=" Expr ";"
```
Присваивание разбирается **вне** Pratt-таблицы, отдельной веткой в `statement()`.
Левая часть (`Expr`, фактически уже распарсенная как произвольное выражение) обязана
после разбора относиться к одному из типов узлов `Identifier`, `Indexing`,
`MemberAccess` — иначе статическая ошибка "Недопустимое выражение слева от оператора
присваивания '='". Другими словами, допустимые lvalue:
```ebnf
LValue ::= IdentifierPath | Postfix "[" Expr "]" | Postfix "." Identifier
```

```ebnf
ExprStmt ::= CallExpr ";"
```
Инструкция-выражение (то есть строка вида `выражение;`, не являющаяся присваиванием)
обязана быть **вызовом функции** (`NodeType::Call` в корне выражения) — любое иное
выражение в этой позиции ловится статической ошибкой "Инструкция-выражение может быть
только вызовом функции" уже после успешного парсинга выражения. Формально это означает,
что постфиксная цепочка `foo().bar` или `foo()[0]` как самостоятельный statement
**не допускается** (корневой узел должен быть именно `Call`, а не `MemberAccess`/
`Indexing` поверх вызова).

### 2.5 Встроенные функции

`печать`, `ввод`, `выход`, `паника`, `утверждение` и т.п. не являются ключевыми словами
лексера. Они синтаксически — обычные идентификаторы, разбираемые как `CallExpr`;
их значение (встроенная функция) разрешается позже, вне парсера.

### 2.6 Выражения (Pratt Parser)

Таблица правил (`get_rule` в `Parser.cpp`) — единственный источник истины по
приоритетам и ассоциативности. Приоритеты перечислены от низшего к высшему:

```
PREC_NONE < PREC_ASSIGNMENT < PREC_OR < PREC_AND < PREC_EQUALITY
         < PREC_COMPARISON < PREC_TERM < PREC_FACTOR < PREC_UNARY
         < PREC_CALL < PREC_PRIMARY
```

⚠ **ТРЕБУЕТ РАЗРЕШЕНИЯ**: уровни `PREC_OR` и `PREC_AND` объявлены в `enum Precedence`,
но в таблице `get_rule` **нет строк** для `AmpAmp` (`&&`), `PipePipe` (`||`) и `Bang`
(`!`) — их поля `prefix`/`infix` остаются `nullptr` (значение по умолчанию для
неинициализированных элементов `std::array<ParseRule, N> r{}`). Аналогично `Percent`
(`%`) не имеет строки. Практический эффект:
- `&&`, `||` как инфиксные операторы **не парсятся**: встретив такой токен в позиции,
  где ожидается конец выражения, `parse_precedence` просто завершит цикл (условие
  `precedence <= get_rule(peek().type)->precedence` не выполнится, т.к. precedence по
  умолчанию — `PREC_NONE`), и токен `&&`/`||` останется неразобранным для вызывающего
  кода (что обычно приведёт к ошибке на уровне `consume(Semicolon, ...)` в
  вызывающей инструкции).
- `!` как префиксный (унарный логический) оператор **не парсится вовсе** — попытка
  начать выражение с `!` вызовет ошибку "Ожидалось выражение" (`prefix == nullptr`).
- `%` как инфиксный оператор **не парсится** — по той же причине, что и `&&`/`||`.

Таким образом, **фактическая грамматика выражений** текущего парсера не содержит
логических операторов и остатка от деления, несмотря на то что они существуют в
лексере (см. также ⚠ 1.4). Ниже приведена именно фактически реализованная грамматика:

```ebnf
Expr        ::= Equality

Equality    ::= Comparison (("==" | "!=") Comparison)*
Comparison  ::= Term (("<" | "<=" | ">" | ">=") Term)*
Term        ::= Factor (("+" | "-") Factor)*
Factor      ::= Unary (("*" | "/") Unary)*

Unary       ::= "-" Unary
              | Postfix

Postfix     ::= Primary PostfixOp*
PostfixOp   ::= "(" Arguments? ")"
              | "[" Expr "]"
              | "." Identifier

Primary     ::= IdentifierPath
              | StructLiteral
              | IntLiteral
              | FloatLiteral
              | StringLiteral
              | BoolLiteral
              | ArrayLiteral
              | CastExpr
              | "(" Expr ")"
```
Все бинарные операторы (`Equality`, `Comparison`, `Term`, `Factor`) — строго
**левоассоциативны**: правый операнд разбирается с приоритетом `rule->precedence + 1`,
левый берётся как уже готовый узел (`nodes.size() - 1`).
Унарный оператор — только `-` (префиксный минус); унарного `+` не существует.
Постфиксные операции `()`, `[]`, `.` имеют одинаковый приоритет (`PREC_CALL`) и
свободно комбинируются в любом порядке и в любом количестве в общем Pratt-цикле —
например, синтаксически допустимо `a[0](x).y[1](z)`, а также вызов результата
скобочного выражения `(f)(x)` или `(a+b)(c)` (что для скобочной группы `grouping`
семантически может быть бессмысленно, но парсером не запрещено).

```ebnf
IdentifierPath ::= Identifier ("::" Identifier)*
```
Разбирается функцией `variable()`: последовательность `Identifier ("::" Identifier)*`
склеивается **в одну строку** (через конкатенацию с `"::"` и повторное интернирование
в `StringPool`) и представляется **единственным** узлом `NodeType::Identifier` с
составным именем-леммой (например, `"Ns::foo"`). Структурного разбора компонентов пути
на уровне AST не производится — разрешение пространств имён полностью текстовое.

```ebnf
StructLiteral ::= IdentifierPath "{" (Expr ("," Expr)*)? "}"
```
Литерал структуры распознаётся, если сразу после `IdentifierPath` следует `{` —
эта эвристика применяется **всегда** внутри `variable()`, то есть после разбора имени
переменной, если далее идёт `{`, парсер безусловно трактует это как литерал структуры
(даже если это, например, начало блока после идентификатора в другом контексте — такой
случай в текущей грамматике statement/expr не встречается, т.к. `{` после голого
идентификатора-выражения не предусмотрен нигде, кроме структурного литерала).

```ebnf
ArrayLiteral ::= "[" (Expr ("," Expr)*)? "]"
```
Вложенные литералы массивов (`[[1,2],[3,4]]`) явно запрещены статической ошибкой
"Вложенные литералы массивов запрещены" — проверяется по наличию `[` сразу после `[`
или `,` внутри `array_literal`.

```ebnf
CastExpr ::= "как" "<" TypeAny ">" "(" Expr ")"
```
Тип внутри `как<...>` — снова `advance()` (см. ⚠ 2.2 п.2), то есть один произвольный
токен без валидации на уровне синтаксиса.

```ebnf
Arguments ::= Expr ("," Expr)*
```
Список аргументов вызова — не менее одного выражения, если скобки не пустые; пустой
список аргументов (`f()`) допустим.

### 2.7 Сводная таблица приоритетов операторов (фактически реализованных)

| Приоритет (низкий → высокий) | Операторы |
|---|---|
| Equality | `==` `!=` |
| Comparison | `<` `<=` `>` `>=` |
| Term | `+` `-` (бинарный) |
| Factor | `*` `/` |
| Unary | `-` (унарный, префикс) |
| Postfix / Call | `()` `[]` `.` |

`=` (присваивание) не входит в эту таблицу — обрабатывается отдельно на уровне
`Stmt` (см. 2.4 `AssignStmt`), а не как оператор выражения.

## 3. Итоговый список мест, требующих разрешения

1. Ключевое слово `массив` объявлено, но не используется — уточнить назначение или
   удалить.
2. Тип `символ`/`Char` объявлен, но лексер не умеет сканировать символьные литералы —
   решить, добавлять ли сканирование `'c'`, либо убрать тип из языка на время MVP.
3. `&&`, `||`, `!`, `%` — токенизируются, но не подключены в таблицу Pratt-парсера;
   решить, требуется ли их поддержка для MVP или это осознанное ограничение.
4. Одиночные `&` и `|` молча теряются лексером без создания токена ошибки — решить,
   нужно ли на них генерировать `TokenType::Error`.
5. Разбор типа (`Type`/`TypeAny`) непоследователен между `VarDecl` (эвристика +
   ограниченный набор ключевых слов/идентификатор) и параметрами функций / полями
   структур / типом возврата / алиасом / cast-выражением (произвольный один токен без
   проверки) — решить, унифицировать ли разбор типов через единую грамматическую
   продукцию `Type`.
6. Массивы поддержаны только для переменных (`VarDecl`); поля структур и параметры
   функций не могут быть массивами — решить, требуется ли расширение для MVP.
7. Квалифицированные имена типов (`Ns::Тип`) не поддерживаются в объявлении переменных
   (только в выражениях через `IdentifierPath`) — решить, нужна ли поддержка для MVP.
