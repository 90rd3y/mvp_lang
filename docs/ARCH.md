# Архитектура интерпретатора (Верхний уровень)

Проект представляет собой модульный интерпретатор языка программирования, построенный по принципу однонаправленного последовательного конвейера (pipeline). 

---

## 1. Концепция конвейера (Pipeline)

Интерпретатор разделен на последовательные фазы обработки. Каждая фаза принимает на вход результат работы предыдущей фазы и не зависит от её внутренней реализации.

```text
  [ Исходный код (.lang) ]
              │
              ▼
         [ Лексер ]          ──► Поток токенов (с IdentId)
              │
              ▼
         [ Парсер ]          ──► Абстрактное синтаксическое дерево (AST)
              │
              ▼
       [ Семантика ]         ──► Аннотированное AST (с TypeId)
              │
              ▼
     [ Интерпретатор ]       ──► Результат выполнения
```

Для обеспечения изоляции каждый этап реализуется в рамках своего пространства имен:
* `Lexer::` — лексический анализатор.
* `Parser::` — синтаксический разбор и построение AST.
* `Semantic::` — семантические проверки и разрешение типов.
* `Interpreter::` — выполнение программы.

---

## 2. Описание фаз обработки

### 2.1. Лексический анализатор (Lexer)
Лексер выполняет первичную обработку текста программы:
* Преобразует исходный текстовый файл в линейный поток токенов.
* Каждый токен содержит информацию о своем типе и точной позиции в кодовой базе (для вывода диагностических сообщений).
* Выполняет интернирование идентификаторов и строковых литералов.
* При обнаружении любой лексической ошибки немедленно выводит диагностическое сообщение и останавливает выполнение.

### 2.2. Синтаксический анализатор (Parser)
Парсер отвечает за разбор структуры программы:
* Преобразует плоский поток токенов в иерархическую структуру — абстрактное синтаксическое дерево (AST).
* Управляющие конструкции, объявления функций и переменных разбираются классическим методом рекурсивного спуска.
* Для выражений (математических, логических и т.д.) применяется специализированный Pratt-парсер.
* Синтаксическое дерево хранится в плоской структуре памяти, узлы используют числовые идентификаторы для связи друг с другом.

### 2.3. Семантический анализатор (Semantic)
Семантический анализатор проверяет логическую корректность программы перед запуском:
* Выполняет разрешение имен (связывание использований переменных и функций с их объявлениями).
* Контролирует области видимости переменных.
* Выполняет проверку типов (type checking), гарантируя отсутствие несовместимых операций.
* Аннотирует AST метаданными о типах, интернируя их.

### 2.4. Выполнение программы (Interpreter)
Финальный этап конвейера, отвечающий за выполнение кода:
* Реализует прямое обходное исполнение (Tree-walking interpreter) на основе аннотированного AST.
* Управляет состоянием программы во время выполнения, включая стек вызовов функций и значения локальных/глобальных переменных.
* Оперирует оптимизированной стратегией выделения памяти в рантайме.
